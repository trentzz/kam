# SV-CLS-001: NovelInsertion Misclassified as TandemDuplication

## Summary

All NovelInsertion (NovIns) variants in the sv_new benchmark suite are
classified as TandemDuplication in the committed results. The root cause is
stale benchmark data: results were generated with a binary built before
NovelInsertion classification existed, and were never regenerated after the
fix landed.

The current code is correct. Rerunning the benchmark with the current
binary will produce the correct NovelInsertion classification.

## Timeline

| Commit | Description |
|--------|-------------|
| `ac85d4f` and earlier | All large insertions (>=50 bp) unconditionally classified as `TandemDuplication`. No `NovelInsertion` type, no `is_tandem_duplication()` function. |
| `dad04a4` | Added `NovelInsertion` variant type, `is_tandem_duplication()` function, and the `classify_variant()` branching logic that distinguishes the two. |
| `1aa26a1` | Added benchmarking infrastructure (scripts, configs, truth VCFs). |
| `87d4bb3` | Committed benchmark results. NovIns results show `TandemDuplication` because the binary used for the run was built from pre-`dad04a4` code. |

## Evidence

### The pre-fix code (before `dad04a4`)

In `classify_variant()`, the `Ordering::Less` branch (alt longer than ref)
was unconditional:

```rust
Ordering::Less => {
    if alt_seq.len() - ref_seq.len() >= SV_LENGTH_THRESHOLD {
        VariantType::TandemDuplication  // <-- always TandemDuplication
    } else {
        VariantType::Insertion
    }
}
```

No `is_tandem_duplication()` function existed. No `NovelInsertion` type
existed. Every large insertion became TandemDuplication regardless of
sequence content.

### The current code (from `dad04a4`)

```rust
Ordering::Less => {
    let ins_len = alt_seq.len() - ref_seq.len();
    if ins_len >= SV_LENGTH_THRESHOLD {
        if is_tandem_duplication(ref_seq, alt_seq) {
            VariantType::TandemDuplication
        } else {
            VariantType::NovelInsertion
        }
    } else {
        VariantType::Insertion
    }
}
```

The `is_tandem_duplication()` function (lines 660-691 of caller.rs):

1. Finds the diverge point between ref and alt.
2. Extracts the inserted bytes from alt.
3. Constructs `ref + ref` (doubled reference).
4. Searches for the inserted bytes as a substring of the doubled reference.
5. Returns true only if the inserted sequence is found.

### Manual trace with benchmark sequences

From the TSV at `sim_novins_vaf0100_a/discovery/variants.tsv`:

- **ref_seq**: 299 bp (the ins_targets.fa target sequence).
- **alt_seq**: 379 bp (walked path including the 80 bp novel insertion).
- **ins_len**: 80 bp.

The novel insertion sequence (from the truth VCF):
```
CCGGCGTCACCCGATGATCAGGTGCCTAGAATCAAGCGTAAGGGCTCGTCGTCACATGTGATAATCTTCCGTGTCCAGCC
```

This sequence was generated from a seeded RNG and verified to share no
20-mer with the 2000 bp synthetic reference or its reverse complement
(see `make_novins_suite.py`).

Tracing through `is_tandem_duplication()`:

1. **diverge = 100**: ref[100] = 'C', alt[100] = 'T'. The sequences match
   for the first 100 bytes, then diverge.
2. **inserted = alt[100..180]**: 80 bytes starting at the diverge point.
   These are `T` + first 79 bp of the novel insert. The offset by one
   occurs because the VCF anchor base (T at position 400) shifts the
   diverge point relative to the target window coordinates.
3. **doubled = ref + ref**: 598 bytes.
4. **Substring search**: the 80-byte inserted sequence is not found
   anywhere in the 598-byte doubled reference.
5. **Result: false**. Classification: NovelInsertion.

This was verified by compiling and running the exact algorithm in both
Python and standalone Rust with the benchmark sequences. Both return
`false`.

### Confirmation that existing unit tests pass

All four existing NovelInsertion tests pass:

- `classify_novel_insertion_random_sequence`: 60 bp of all-C inserted into
  ACGT repeats. Classified as NovelInsertion.
- `classify_tandem_dup_stays_tandem_duplication`: 60 bp copied from ref
  inserted back. Classified as TandemDuplication.
- `classify_novel_insertion_at_sv_threshold`: 50 bp (exact threshold) of
  all-T. Classified as NovelInsertion.
- `integration_novel_insertion_full_pipeline`: full call pipeline with
  NovelInsertion, including filter checks.

## Root cause

The benchmark results at commit `87d4bb3` were generated by a Claude agent
running `kam run` with a binary built from code that predated the
NovelInsertion classification (`dad04a4`). In that older code, all large
insertions were unconditionally classified as TandemDuplication.

The benchmarking infrastructure (`run_sv_new_suite.sh`, added in `1aa26a1`)
was committed AFTER the result data. Its output directories (`kam_*`) do
not exist, confirming it was never run.

The committed result data lives in `sim_*/discovery/` and
`sim_*/tumour_informed/` subdirectories, placed there directly by the
agent's `kam run` invocations.

## Why existing tests did not catch this

The unit tests for NovelInsertion classification use synthetic sequences
(ACGT repeats with all-C or all-T inserts). They correctly verify the
`is_tandem_duplication()` logic. The tests pass because the logic IS
correct.

The problem is not in the classification logic. It is in the benchmark
pipeline: stale results from a pre-fix binary were committed and never
regenerated.

No integration test exists that runs `kam run` end-to-end on the NovIns
varforge data and checks the variant_type column.

## Proposed fix

1. **Regenerate benchmark results**: Run `run_sv_new_suite.sh` (or the
   equivalent `kam run` commands) with the current binary for all NovIns
   samples. The current `is_tandem_duplication()` logic will correctly
   classify them as NovelInsertion.

2. **Add a regression test**: Add a unit test in caller.rs using the exact
   ref_seq and alt_seq from the benchmark TSV to lock in the correct
   classification:

   ```rust
   #[test]
   fn classify_benchmark_novins_as_novel_insertion() {
       let ref_seq = b"AACAGCTCAGG..."; // 299 bp from ins_targets.fa
       let alt_seq = b"AACAGCTCAGG..."; // 379 bp from benchmark
       assert_eq!(
           classify_variant(ref_seq, alt_seq),
           VariantType::NovelInsertion,
       );
   }
   ```

3. **Re-score and update figures**: After regenerating results, re-run the
   scoring and plotting scripts to update the sensitivity curves. The
   NovIns sensitivity numbers should remain the same (detection is
   unaffected), but the variant_type column will change from
   TandemDuplication to NovelInsertion.

## Note on the `is_tandem_duplication()` logic

The function is correct for its intended purpose. However, the diverge-point
extraction is slightly offset from the VCF insertion coordinates. When the
VCF anchor base at the insertion site differs from the reference base at
the same position in the target window, the algorithm extracts `inserted`
bytes that are shifted by one relative to the actual novel insert sequence.

For the NovIns benchmark case:
- The actual novel insert: `CCGGCGTCACCCG...TCCAGCC` (80 bp)
- What the algorithm extracts: `TCCGGCGTCACCC...TCCAGC` (80 bp, shifted by 1)

This shift does not affect correctness. The shifted extracted bytes are
equally absent from the doubled reference, so the function returns the
correct answer. However, if future work needs to compare the extracted
inserted sequence to a known insert (for reporting or matching purposes),
this offset should be accounted for.
