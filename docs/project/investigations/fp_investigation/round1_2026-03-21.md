# FP Investigation Round 1 — 2026-03-21

## Summary

This document covers the first round of false positive (FP) investigation for the `kam` variant caller.
The investigation identified the primary FP categories in the 30ng/2% VAF condition, motivated
structural code changes in v5, and benchmarked the result. Precision did not improve in v5 because
the changes were correctness fixes rather than filtering changes.

---

## Root Cause Analysis (v4 baseline)

Condition: 30ng, 2% VAF, 2M reads. v4 results: 224 TP, 92 FP, precision = 0.709.

FPs were classified by VAF and molecule count:

| Category | Count | VAF range | Notes |
|---|---|---|---|
| High-VAF (germline) | ~10 | 0.47–1.0 | Heterozygous germline variants in ctDNA |
| n_alt = 2 | ~38 | typically 0.01–0.05 | Likely artefacts or rare cfDNA biological variants |
| n_alt = 3–9 | ~34 | 0.01–0.03 | Mix of artefacts and real low-VAF cfDNA variants |
| n_alt ≥ 10 | ~10 | > 0.03 | Biological cfDNA variants not in the spike-in truth set |

Total: ~92 FPs.

Key insight: most low-n_alt FPs (n_alt = 2–9) likely represent real biological variants in the
cfDNA background that are not in the Twist spike-in truth set. They are not sequencing errors:
they pass the strand bias filter and have posterior confidence > 0.99. Distinguishing them from
true somatic variants is fundamentally hard without orthogonal evidence (e.g., matched normal).

### Why the LowDuplex filter was not firing (v4 bug)

In v4, `n_duplex_alt` was computed as:

```rust
let n_duplex_alt = alt_evidence.mean_duplex.round() as u32;
```

`mean_duplex` is the mean duplex count across all ~70 k-mers in the alt path. The majority of
these k-mers are shared with the reference path (anchor k-mers), where many reference-duplex
molecules overlap. This made `mean_duplex` always ≥ 30–90 even for low-evidence variants.
The `LowDuplex` filter (`min_alt_duplex = 1`) never triggered.

---

## v5 Code Changes

Three changes were made to correct the duplex computation and add new filtering options.

### 1. Variant-specific duplex (`min_variant_specific_duplex`)

Added a new field to `PathEvidence` and `ScoredPathRecord`:

- `min_variant_specific_duplex: u32` — the minimum duplex count at k-mers that are present in
  the alt path but absent from the reference path's k-mer set.
- Computed in `score_and_rank_paths` (kam-pathfind) by building a `HashSet<u64>` of canonical
  reference k-mers and filtering to alt-only k-mers.
- Propagated through the bincode pipeline to the caller.
- Used in `call_variant` as `n_duplex_alt` (replacing the broken `mean_duplex` field).

### 2. Single-molecule duplex exception

Added `min_alt_duplex_for_single: Option<u32>` to `CallerConfig`. When set (default: `Some(1)`),
a call with n_alt = 1 passes the molecule threshold if its variant-specific duplex count is
≥ the threshold. This allows rare, low-coverage targets (e.g., EGFR T790M, n_ref = 13) to
be called from a single duplex-confirmed molecule.

### 3. max_vaf filter

Added `max_vaf: Option<f64>` to `CallerConfig`. When set, calls with VAF > max_vaf are
assigned the `HighVaf` filter label. Default: `None` (disabled). Intended to reject germline
heterozygous variants (VAF ≈ 0.5) in ctDNA somatic variant calling.

### Summary

v5 introduced 2 TPs and 33 FPs net across all 24 samples.

Most of the added FPs occur at 15ng/0.01% VAF (+10) and 5ng/0% VAF (+4). These are from the
single-molecule duplex exception: n_alt=1 calls that are duplex-confirmed now pass rather than
being filtered as LowConfidence. At very low VAF or 0% VAF, these are almost entirely FPs.

The variant-specific duplex fix (replacing `mean_duplex` with `min_variant_specific_duplex`)
did not change call outcomes because the default `min_alt_duplex = 0` disables the LowDuplex
filter. Enabling `min_alt_duplex = 1` removes too many TPs (~36% of genuine 2% VAF variants
have zero variant-specific duplex at 6.6% duplex rate), so the filter cannot be enabled at
current sequencing depths.

---

## Why Precision Did Not Improve

The v5 changes were structural (correct the broken duplex field, expose new filter options) but
did not add any filtering that was enabled by default. The max_vaf filter was added as an option
but not set in the default config.

The single-molecule exception marginally increased FPs without equivalent TP gains overall.

---

## Proposed Round 2 Changes

1. **Enable max_vaf via CLI flag.** Add `--max-vaf` to both `CallArgs` and `RunArgs`. Run the
   benchmark with `--max-vaf 0.35` to remove germline heterozygous FPs.

2. **Evaluate min_alt_molecules = 3.** Raising the molecule threshold from 2 to 3 would remove
   ~38 n_alt=2 FPs at the cost of some low-count TPs. Benchmark both thresholds to quantify
   the TP/FP trade-off.

3. **Disable single-molecule exception at low coverage.** The `min_alt_duplex_for_single`
   exception adds FPs at low VAF. Consider restricting it to samples with ≥ 10% duplex rate.

These three changes are expected to meaningfully reduce FP count at the benchmark conditions
most relevant to clinical use (30ng/1–2% VAF).
